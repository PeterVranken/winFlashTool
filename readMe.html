This repository provides a CAN-based flash tool for embedded devices. The tool uses CCP as the communication protocol with a flash bootloader (FBL) running on the embedded target. CCP commands are used to erase the flash ROM and to download new firmware.

The tool has been developed and tested on Windows. It supports the inexpensive PEAK PCAN-USB adapter as CAN hardware. The implementation is written in Java, and since PEAK drivers and JNI libraries are available for Linux, the flash tool should in principle work there as well, although this has not yet been tested.

A related repository contains an FBL for the NXP MPC5748G MCU, which works with this flash tool out of the box. Other FBL implementations will likely require adjustments either in the FBL or in this tool to ensure mutual compatibility.

1. Connection test with DEVKIT-MPC5748G

A basic connection test can be performed using the NXP DEVKIT-MPC5748G board. A compatible flash bootloader is available. Make sure the FBL has been flashed onto the board beforehand.

Connect a PEAK PCAN-USB adapter to a USB port of your Windows PC. Then connect the CAN plug of the PCAN-USB to plug P5 on the DEVKIT-MPC5748G (taking termination into account).

Open a PowerShell window and verify that a suitable Java JDK is available:

java -version
javac -version

The version should be Java 18 or newer. Older versions may work but have not been tested.

Build the flash tool:

cd <repoCloneDirectory>
.\gradlew installDist

Display the flash tool’s help text and then enumerate the available, connected PEAK PCAN-USB adapters:

.\winFlashTool.ps1 --help
.\winFlashTool.ps1 --enumerate-CAN-devices

The following steps require communication with the FBL. They will work only if the FBL was built with its default configuration (as provided in the GitHub repository):

  • CAN Classic with 500 kBd.

  • The default authentication key pair is used.

  • The CAN IDs for CCP CRO and DTO messages are 0x600 and 0x650, respectively.

  • The CCP station address is 0x1234, or 4660.

  • DEVKIT-MPC5748G and the PCAN-USB adapter are the only connected CAN nodes.

Ensure that the DEVKIT-MPC5748G is powered off or held in reset (press and hold the push button RESET). Now start the flash tool so it waits for the FBL to start after a delay:

.\winFlashTool.ps1 --station-address 4660 --ignore-ack-err-for-connect --key-file .\privateKey.sk --upload-version-fbl

Note, the same command can be abbreviated by using the short form of the command line switches:

.\winFlashTool.ps1 -a 4660 -ie -sk .\privateKey.sk -uv

Shortly after running one of these commands, start the DEVKIT-MPC5748G. Release the reset button or connect the USB cable. You have about 20 seconds to start the FBL.

The flash tool should then print the FBL version information in its console output. You should get something like:

If the expected output does not appear, you have two initial debugging options:

  1. Add --log-level DEBUG or --log-level TRACE to the flash tool command line and repeat the procedure. This provides more detailed diagnostics.

  2. Open a terminal application, connect to the COM port of the DEVKIT-MPC5748G, and repeat the procedure. See its readme for instructions. This reveals how the FBL reacts and responds.

If you have just flashed the FBL onto the DEVKIT-MPC5748G but have not yet flashed an application, then neither the reset button nor delayed board power-up is required. The FBL starts immediately after reset and remains active, listening for a CAN CCP connection at all times.

If this connection test succeeds, then downloading an application, uploading memory, or erasing flash should work without difficulty.

As a last example: Download and flash the sample application from the FBL repository, upload it again for verification and eventually reset the board in order to start the flashed application:

$fblDir = "<cloneDirOfFblRepository>"
$app = "$fblDir\applications\startup\bin\ppc\PRODUCTION\DEVKIT-MPC5748G-startup.s28"
.\winFlashTool.ps1 -m MPC5748G -a 4660 -sk privateKey.sk -ie -i $app -r APP

2. Command line switches of the flash tool

Common syntax rules:

  • A command line switch and its value or argument must always be separated by a blank. A form such as --log-level=OFF is not recognized.

  • Switches that do not require a value or argument must not be combined. A construct like -de is not interpreted as -d -e.

-kp

The tool exits after generating the key pair.

Optional, default is not to generate a key pair.

3. To verify or not to verify

The flash tool has the command line switch --no-verify. Normally, when downloading data to the ECU for programming, the procedure has two phases. In phase one, all data from the srec input file is downloaded and programmed and in phase two, the programmed address space is uploaded from the ECU and compared to the contents of the same srec file. Mismatches would be reported as verify errors. Switch --no-verify allows to omit the second phase.

The upload phase is even a bit slower than the download and program phase. If you use the flash tool with the FBL from the related repository, it may not even be very useful; this FBL implicitly verifies every program operation immediately after programming. A later upload and data comparison by the flash tool has the only added value of verifying the CAN data transmission — but at the cost of doubling the flash time. CAN errors are rare, so most of the time during application development, this time could be saved using --no-verify. (For end-of-line flashing in a productive process, verification is of course recommended.)

If doubts should arise later about the error-free flashing process, the omitted verification step can always be performed later. Switch --srec-input-file in combination with --verify-only starts the procedure to only do the data update and comparison. The verify result is finally printed in the log.

4. Connecting to the FBL

There are several ways to establish the connection between the flash tool and the FBL. Some of these are application-specific; the flashed application has one or more trigger options to switch from the running application to the FBL, which is then available for CCP communication. These solutions are application-specific and will not be discussed further here.

Furthermore, many FBLs share the characteristic that, regardless of these specific solutions, they allow contact with the flash tool immediately after each reset. The MCU reset starts the FBL, which then waits a short time for a possible connection via CAN and CCP. Only if this time has elapsed without a connection being established does the FBL start the flashed application. This procedure is a backdoor to allow reprogramming of the application even if the already flashed application has an error and the normally used, specific solution therefore fails.

Unconditionally waiting for a possible connection means an undesirable increase in the application’s startup time. This time is therefore kept to a minimum. This, in turn, makes it difficult to start the flash tool at precisely the right point. Our flash tool offers two options to simplify the process.

4.1. Flash tool and target board are the only CAN nodes

If the flash tool is connected to the board running the FBL via a direct point-to-point connection, meaning there are no other CAN communication participants besides these two, the --ignore-ack-err-for-connect option can be used.

With this option, the flash tool is started while the board with the FBL is still inactive. (It may be helt in reset or still switched off.) The flash tool sends the CCP CONNECT message. This message remains unheard on the CAN bus because there is (yet) no other active node that would send the acknowledgment. In this situation, CAN transceivers (such as those used in the PEAK PCAN-USB) continuously resend the message until an acknowledgment is finally received. At the same time, the situation is reported back as a CAN protocol error. The flash tool sees the error but ignores it due to the --ignore-ack-err-for-connect option. Normally, such an error would abort the flashing process, but now the tool simply continues waiting for a response from the FBL.

The board is switched on with a slight delay. The FBL starts, initializes the CAN hardware, and then receives the next repetition of the CCP CONNECT message. Simultaneously, the CAN hardware sends the acknowledgement. The FBL evaluates the message and sends the response — the connection is established.

The maximum time the flash tool waits for the CONNECT response is configurable; the default value is 20 seconds. The board with the FBL must be powered on within this timeframe after the flash tool starts.

4.2. Flash tool and target board are connected to a CAN bus

If the flash tool and board are connected to a CAN bus that also contains other CAN devices, --ignore-ack-err-for-connect will not help. Again, the idea is to start the flash tool first and then, after a short delay, the board with the FBL. The situation is now significantly different; the other CAN nodes would send the acknowledgement as soon as the CCP CONNECT message is sent for the first time. The flash tool would assume that the CONNECT was understood and would be answered shortly afterward — which doesn’t happen because the board with the FBL is still powered off. The flash tool would then abort the flashing process with an error message.

In this situation, the other option --no-retries-ccp-connect helps. If a CCP CONNECT is not answered promptly, the flash tool sends a repeated CONNECT after a short wait, as many times as the option specifies. If the board with the FBL is powered on after a short delay, the next of these repeated send attempts should fall within the small time window that the FBL maintains for establishing contact. The FBL responds normally, and the connection is established.

The flash tool repeats sending the CCP CONNECT messages approximately every 30 ms. Therefore, the time window in the FBL should be configured to approximately 50 ms (or more) to guarantee that at least one flash tool send repetition is seen.

Please note that the flash tool’s 30 ms repetition time is a compile-time constant and cannot be changed via command-line arguments.

The default value for --no-retries-ccp-connect is 20. If you can reliably start flash tool and board within half a second, this option is not needed.

5. Typical modifications of the flash tool

It is unlikely that you can use the flash tool without any modification. The tool works out of the box only if:

  • You use an MPC5748G board together with the FBL from the related repository.

  • You migrate the FBL from the related repository to one of the similar MCUs MPC5775B/E or MPC5777C and use it.

  • You have your own FBL implementation that uses CCP and supports the same CCP commands in the same manner.

In all other situations, some level of adaptation will be required. The smallest necessary modification is typically adding support for another MCU type.

5.1. Add another MCU’s memory map

The flash tool is aware of the target MCU only to the extent needed to optimize CCP erase operations. Erase commands are derived from the flash block layout of the MCU. S-records in the input file refer to specific flash regions, and only those regions containing at least one byte to be programmed are erased. Knowledge of the flash block structure is also used to validate the S-record input: any S-record referencing an address outside the supported flash range results in an early error, even before any communication with the ECU begins.

To add support for a new MCU, create a memory map class derived from winFlashTool.mcu.Flash. Use Mpc5748G_C55FMC.java as a template; you mainly need to fill in a table describing the addresses of the flash blocks managed by your FBL.

To make the new MCU selectable, extend the command-line handling in WinFlashTool.java:

  1. In method defineArguments, update the help text for the --mcu-target option so your new MCU appears in the list of supported devices.

  2. In method run (around line 675), a switch/case statement creates the appropriate Flash object based on the selected MCU. Add a case for your new MCU and instantiate your new memory map class.

6. Building and deploying the flash tool as standalone application

6.1. Building the flash tool

The build of the Java application is under control of Gradle. After cloning the repository, you’d open a shell window (e.g. PowerShell) and cd into the root of the chosen clone directory:

cd <cloneDirOfFlashToolRepository>
gradlew installDist

The very first build may take several minutes. Gradle will download all if its own libraries from the Internet and all libraries required for the flash tool. (Subsequent builds are much faster as this doesn’t need to be repeated.)

The Gradle task installDist yields the folder build\install\winFlashTool. This folder is the ready to use bundle of jar files and other libraries plus the inevitable Java launch script. If you followed the instruction given in Section 1 and ran the flash tool using winFlashTool.ps1 then you effectively ran the Gradle created Java launch script build\install\winFlashTool\bin\winFlashTool.bat; the PowerShell script is a one-liner for calling the script generated by Gradle.

6.2. Deploying the flash tool

The Gradle build has created the ready-to-use bundle of files, directly usable as folder build\install\winFlashTool. You can copy this folder anywhere, e.g., to C:\Program Files\.

Alternatively, you can run this Gradle command:

gradlew distZip

It creates the same bundle of needed files as ZIP archive build\distributions\winFlashTool-<version>.zip. You can distribute it and extract the contents at any location, e.g., C:\Program Files\.

Using the flash tool means running the launch script in folder <chosenTargetDir>\winFlashTool\bin (or <chosenTargetDir>\winFlashTool-<version>\bin if you extracted the ZIP archive). To make this script visible and accessible from any working directory, you should add this folder to the Windows search PATH. To check the installation, open a new shell window (e.g., PowerShell) and type:

where.exe winFlashTool

The response should point to your chosen installation folder. Now try:

winFlashTool --help
